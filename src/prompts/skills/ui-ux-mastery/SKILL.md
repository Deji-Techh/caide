# World-Class UI/UX Production Skill

## 0. Mission

Create interfaces that are:

- immediately understandable;
- recognizably appropriate for the product and its users;
- visually distinctive without becoming theatrical or confusing;
- complete across states, devices, inputs, and failure conditions;
- accessible by default;
- feasible to implement in the selected technology stack;
- coherent as a system rather than impressive only in screenshots;
- validated against explicit quality gates before delivery.

This is not a style catalogue. It is an operating system for design work.

The agent must not merely produce attractive screens. It must reason from user goals, product constraints, information architecture, interaction models, content, technical constraints, and risk. Visual treatment follows those decisions.

---

## 1. Activation Criteria

Use this skill whenever the task includes one or more of the following:

- designing a new interface, page, app, flow, component, dashboard, website, or product;
- redesigning an existing interface;
- converting requirements, screenshots, wireframes, or product notes into UI;
- creating a design system or component library;
- generating production frontend code where visual and interaction quality matter;
- auditing usability, accessibility, visual hierarchy, responsiveness, or consistency;
- defining UX flows, navigation, information architecture, onboarding, forms, or empty states;
- designing AI assistants, editors, IDEs, creator tools, finance tools, health tools, marketplaces, social products, or operational software;
- translating a visual direction into implementation specifications;
- reviewing an existing repository and improving its UI/UX without breaking functionality.

Do not use this skill as a substitute for domain expertise, user research, security review, legal review, medical review, financial compliance, or safety engineering. Apply it alongside those disciplines.

---

## 2. Instruction Priority

When requirements conflict, use this priority order:

1. Safety, law, privacy, and explicit user constraints.
2. Correctness of the product task and preservation of critical functionality.
3. Accessibility and inclusive operation.
4. User comprehension and task completion.
5. Platform conventions and technical feasibility.
6. Information architecture and interaction coherence.
7. Performance and reliability.
8. Visual hierarchy and brand expression.
9. Motion, polish, and delight.
10. Novelty.

Never sacrifice comprehension, accessibility, or task completion merely to create a more dramatic visual.

## 3. Requirement Strength

Interpret rules using these levels:

### MUST

A non-negotiable condition. Violating it requires a documented reason and an alternative safeguard.

### SHOULD

A strong default. Deviate only when product context clearly supports another choice.

### MAY

An optional technique. Use only when it improves the experience.

### NEVER BY DEFAULT

A pattern that commonly produces weak or generic results. It may be used only when the product, brand, content, and context justify it.

## 4. Core Operating Principles

### 4.1 Outcome before interface

Users do not want screens. They want outcomes. Identify the desired outcome before selecting layout, components, visual style, or technology.

For every major screen, answer:

1. Who is using this?
2. What are they trying to accomplish now?
3. What information do they need before acting?
4. What is the primary action?
5. What can go wrong?
6. How do they recover?
7. What must remain visible while they work?

### 4.2 Product-specific, not template-specific

The design must reflect the product's domain, frequency of use, risk, content density, platform, user sophistication, and emotional context.

Do not reuse the same visual grammar for unrelated products. A trading terminal, meditation app, code editor, hospital portal, and fashion storefront require different densities, interaction patterns, and trust signals.

### 4.3 Clarity before cleverness

A user should understand where they are, what they can do, and what happens next without decoding decorative metaphors.

Novelty is valuable only when it does not increase task uncertainty.

### 4.4 Recognition before recall

Expose relevant options, context, status, history, and constraints. Do not force users to remember information from previous screens when it can be shown or carried forward.

### 4.5 Progressive disclosure

Keep the initial view focused. Reveal advanced controls when context, permission, expertise, or task state makes them relevant.

Do not hide critical information under progressive disclosure.

### 4.6 Forgiveness and reversibility

Prefer undo, version history, drafts, autosave, previews, confirmation for high-impact actions, and explicit recovery paths.

Do not use confirmation dialogs for every low-risk action. Confirmation fatigue weakens protection.

### 4.7 Content is part of the interface

Labels, examples, descriptions, values, timestamps, units, table columns, error messages, and empty-state copy determine usability. Do not design around lorem ipsum or meaningless placeholder metrics when realistic content can be inferred.

### 4.8 Accessibility from the first decision

Accessibility is not a final checklist. It affects layout, semantics, component choice, input methods, motion, copy, color, focus order, error handling, and responsive behavior.

Target WCAG 2.2 AA for web interfaces unless a stricter standard is required. Treat platform accessibility guidance as an additional constraint, not a replacement.

### 4.9 Performance is UX

Fast feedback, stable layout, appropriate loading states, efficient assets, and resilient interaction are design requirements.

### 4.10 System over screenshot

A polished hero screen is not a finished product. Define repeated patterns, edge cases, responsive behavior, complete states, component contracts, and implementation logic.

## 5. Anti-Generic Design Policy

The agent must actively prevent generic AI-generated interface output.

### 5.1 Never default to these patterns

- purple-to-blue gradients as the primary identity;
- glassmorphism without a functional layering reason;
- large blurred decorative blobs;
- every section inside a rounded card;
- excessive 16-24px corner radii on all surfaces;
- three identical feature cards with icon, heading, and two lines of text;
- giant empty hero sections with vague claims;
- gradient text used as a substitute for hierarchy;
- random dashboard statistics not required by the product;
- decorative charts with fabricated data;
- pill-shaped controls for every action;
- icon-only navigation where labels are necessary;
- excessive shadows and floating surfaces;
- generic copy such as “Unlock the power of”, “Revolutionize your workflow”, or “Seamlessly transform”;
- fake testimonials, fake company logos, fake user counts, or invented trust claims;
- unnecessary floating action buttons on desktop;
- center-aligned paragraphs longer than two short lines;
- identical spacing and density across all screen regions;
- a sidebar plus four KPI cards plus a large chart merely because the product is called a dashboard;
- animation on every element;
- mobile layouts that are desktop screens squeezed into one column.

### 5.2 Distinctiveness test

Before finalizing a direction, ask:

- Could this design be used unchanged for five unrelated products?
- Does the visual language reveal something specific about the product?
- Are the layout and interaction patterns derived from actual tasks?
- Is there at least one memorable but useful design idea?
- Does the interface still feel coherent when branding is removed?

If the answer to the first question is yes, revise the design.

### 5.3 Originality without imitation

Do not copy a named product pixel-for-pixel. It is acceptable to analyze conventions, strengths, and interaction patterns, but the result must be adapted to the current product and expressed through an original system.

## 6. Full Production Workflow

Follow this sequence unless the user explicitly requests only one phase.

```text
Understand → Classify → Model users and tasks → Define information architecture
→ Generate directions → Select a system → Specify flows and states
→ Design screens and components → Implement or hand off
→ Inspect → Test → Score → Revise → Deliver
```

Do not jump directly from a one-sentence prompt to high-fidelity styling unless the task explicitly asks for visual exploration only.

## 7. Phase 1 — Understand the Assignment

### 7.1 Extract explicit requirements

Capture:

- product name and purpose;
- target users;
- platform: web, iOS, Android, desktop, tablet, TV, spatial, kiosk, embedded;
- required screens and flows;
- primary features;
- brand constraints;
- technical stack;
- existing design system or repository;
- accessibility target;
- supported languages and regions;
- data sensitivity and risk;
- delivery format;
- timeline or implementation scope;
- user-provided references.

### 7.2 Infer cautiously

When information is missing and the task must proceed, make conservative assumptions. Mark high-impact assumptions visibly in the design brief.

Do not invent:

- legal claims;
- certifications;
- customer numbers;
- performance claims;
- prices;
- regulated advice;
- data the product cannot realistically access;
- features that contradict the provided scope.

### 7.3 Define success

Translate vague goals into observable outcomes.

Weak:

> Make it premium and easy to use.

Stronger:

> A first-time user can understand the product value in five seconds, create a project in under one minute, distinguish primary from secondary controls, and complete the core task without reading documentation.

### 7.4 Identify constraints

Constraints may include:

- small screens;
- low bandwidth;
- touch-only use;
- intermittent connection;
- one-handed operation;
- high information density;
- expert workflows;
- auditability;
- irreversible actions;
- multiple roles and permissions;
- legacy systems;
- localization;
- accessibility technology;
- older hardware;
- large datasets;
- real-time updates.

Use constraints to shape the design rather than treating them as cleanup work.

## 9. Phase 3 — Model Users, Jobs, and Context

### 9.1 Define primary users

For each significant user group, identify:

- knowledge level;
- goals;
- frequency of use;
- device and environment;
- time pressure;
- accessibility needs;
- trust concerns;
- consequences of error;
- permissions;
- common interruptions;
- vocabulary.

Avoid fictional demographic detail that does not change design decisions.

### 9.2 Jobs-to-be-done statement

Use this structure:

> When [situation], the user wants to [motivation/task], so they can [desired outcome], while avoiding [risk or frustration].

### 9.3 Task hierarchy

Classify tasks:

- **Primary:** the core reason the product exists.
- **Frequent:** repeated often and requiring efficiency.
- **Critical:** low-frequency but high-impact.
- **Supporting:** required to complete primary tasks.
- **Administrative:** settings, billing, permissions, profile.
- **Recovery:** undo, restore, reset, troubleshoot, contact support.

Design navigation and emphasis based on this hierarchy.

### 9.4 User-state model

Consider differences between:

- first-time and returning users;
- novice and expert users;
- anonymous and authenticated users;
- free and paid users;
- viewer, editor, manager, and administrator roles;
- online and offline users;
- users with no data, partial data, or extensive data;
- users in good standing, restricted status, or failed verification.

## 10. Phase 4 — Information Architecture

### 10.1 Inventory content and actions

List all objects, object attributes, actions, statuses, relationships, settings, and help content.

Example object model:

```text
Project
├── title
├── owner
├── collaborators
├── status
├── modified time
├── files
├── activity
├── permissions
└── actions: open, rename, duplicate, archive, delete, share
```

### 10.2 Choose an architecture pattern

- **Hierarchical:** categories and subcategories.
- **Hub-and-spoke:** central home with task destinations.
- **Sequential:** onboarding, checkout, setup, application.
- **Matrix:** users move among multiple related work areas.
- **Object-oriented:** navigation around entities such as projects, customers, orders, or files.
- **Canvas-centric:** a central work surface surrounded by tools.
- **Feed-centric:** chronological or ranked stream with creation and filtering.
- **Search-first:** large collection where direct retrieval is primary.

### 10.3 Navigation rules

The user must always be able to determine:

- current location;
- parent location;
- available destinations;
- current object or workspace;
- unsaved or pending state;
- how to return safely.

### 10.4 Navigation selection

Use:

- top navigation for a small number of broad destinations;
- sidebar navigation for complex applications with persistent sections;
- bottom navigation for three to five primary mobile destinations;
- tabs for peer views of the same object or context;
- breadcrumbs for deep hierarchical navigation;
- command palettes for expert acceleration, not as the only discovery mechanism;
- contextual menus for secondary object actions;
- search when the content set exceeds comfortable browsing.

Do not use tabs as general site navigation when each tab represents a separate product area.

### 10.5 Label quality

Labels must be:

- specific;
- concise;
- written in user vocabulary;
- distinct from neighboring labels;
- action-oriented for controls;
- consistent across navigation, headings, buttons, and confirmation messages.

Avoid “Manage”, “Solutions”, “Resources”, or “More” when a more specific label exists.

## 11. Phase 5 — Map Flows Before Styling

For each critical flow, define:

1. Entry points.
2. Preconditions.
3. Steps.
4. Decisions.
5. Data required.
6. Validation.
7. Permission or authentication interruptions.
8. Loading and asynchronous behavior.
9. Success state.
10. Failure states.
11. Recovery.
12. Exit and return behavior.
13. Analytics or audit events where relevant.

### 11.1 Minimize unnecessary steps

Every step must do at least one of the following:

- collect required information;
- make a meaningful decision;
- explain risk or consequence;
- establish trust;
- verify identity or permission;
- preview or confirm a consequential result.

Remove steps that exist only because the internal system is poorly organized.

### 11.2 Preserve context

When users move between steps, retain:

- entered values;
- filters;
- selection;
- scroll position where useful;
- draft state;
- source context;
- navigation history.

### 11.3 Avoid dead ends

Every empty, error, denied, expired, or completed state must provide an appropriate next action.

## 12. Phase 6 — Generate Visual Directions

Before selecting a final high-fidelity direction, produce at least two materially different directions when scope permits.

Each direction must define:

- concept and rationale;
- intended emotional quality;
- density;
- typography;
- color logic;
- surface and elevation model;
- corner and shape language;
- imagery or illustration approach;
- iconography;
- motion behavior;
- signature interaction or compositional idea;
- risks and trade-offs;
- product fit.

### 12.1 Example direction families

Use these as starting points, not templates:

- **Editorial precision:** typographic hierarchy, generous rhythm, restrained surfaces.
- **Technical cockpit:** dense panels, strong alignment, compact controls, visible state.
- **Quiet luxury:** restrained palette, precise spacing, high-quality imagery, minimal ornament.
- **Industrial utility:** strong structure, direct labels, durable controls, low decoration.
- **Playful consumer:** expressive shape and motion, friendly copy, simplified decisions.
- **Data-forward analytical:** comparison-first layout, disciplined color, dense but readable tables and charts.
- **Canvas-first creator:** dark or neutral chrome, central work surface, non-destructive tools.
- **Monochrome system:** contrast, typography, texture, and spacing create hierarchy with minimal color.
- **Spatial soft:** layered depth, gentle transitions, large touch-safe controls, calm surfaces.
- **Neo-brutalist controlled:** high contrast, visible structure, sharp typography, deliberate roughness without usability loss.

### 12.2 Select a direction using evidence

Score each direction on:

- product fit;
- user fit;
- brand fit;
- clarity;
- scalability;
- accessibility;
- implementation feasibility;
- distinctiveness;
- content compatibility.

Do not select solely because it looks more fashionable.

## 13. Design-System Foundation

Create or align to a system before designing many screens.

### 13.1 Token layers

Use three layers:

1. **Primitive tokens:** raw values such as neutral-900, blue-600, 8px.
2. **Semantic tokens:** purpose such as text-primary, surface-raised, action-primary.
3. **Component tokens:** local decisions such as button-primary-bg-hover.

Components should consume semantic or component tokens, not arbitrary primitive values.

### 13.2 Required token categories

- color;
- typography;
- spacing;
- size;
- radius;
- border;
- elevation;
- opacity;
- motion duration;
- motion easing;
- z-index;
- breakpoints or container queries;
- data visualization colors;
- focus treatment.

### 13.3 Token naming

Names must describe meaning, not visual appearance alone.

Prefer:

```text
color.surface.canvas
color.surface.raised
color.text.primary
color.text.muted
color.border.subtle
color.action.primary.default
color.status.negative.surface
```

Avoid component code full of values such as `gray-500`, `blue-600`, and `rounded-xl` when those choices represent product semantics.

### 13.4 Density modes

Professional products may require comfortable and compact density modes. Define which dimensions change:

- row height;
- control height;
- internal padding;
- gap;
- type size where appropriate;
- icon size;
- visible metadata.

Do not simply scale the entire interface down.

### 13.5 Theme support

When light and dark themes are required:

- preserve semantic hierarchy rather than invert colors mechanically;
- retune saturation and contrast;
- verify status colors independently;
- use lighter surfaces to represent elevation in dark themes when appropriate;
- test images, charts, borders, disabled states, code blocks, and focus rings;
- avoid pure black or pure white as an automatic rule; use them only when the visual system and comfort support them.

Read `references/design-system.md` for deeper guidance.

## 14. Layout and Spatial Composition

### 14.1 Establish a spatial logic

Use a consistent base unit, commonly 4px, with a practical spacing sequence such as:

```text
0, 2, 4, 6, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96, 128
```

The scale is a system, not a command to use every value.

### 14.2 Relationship before arithmetic

Spacing communicates relationship:

- tighter spacing = stronger relationship;
- larger spacing = separate group or change in subject;
- repeated spacing = repeated structure;
- asymmetrical spacing can create emphasis when intentional.

### 14.3 Grid choice

Choose based on content:

- marketing and editorial: fluid columns with readable line length;
- dashboards: stable grid, predictable alignment, resizable regions when needed;
- forms: usually single-column with grouped sections;
- creator tools: central canvas plus persistent or collapsible tool regions;
- mobile: content stack with safe-area and thumb-zone awareness;
- tables: horizontal space prioritized for comparison.

A 12-column grid is a useful tool, not a universal requirement.

### 14.4 Reading width

For sustained body reading, generally target 45–80 characters per line. Use wider measures only for content that benefits from scan patterns, tables, code, or side-by-side comparison.

### 14.5 Hierarchy check

Every screen should contain:

- one dominant context or title;
- one clear primary task region;
- grouped secondary information;
- subdued tertiary metadata;
- visible navigation and status;
- intentional whitespace or density.

### 14.6 Responsive transformation

Responsive design is not shrinking. Define how each region:

- reflows;
- collapses;
- becomes a drawer or sheet;
- changes order;
- hides nonessential detail;
- preserves critical actions;
- handles overflow;
- changes interaction method;
- supports safe areas and virtual keyboards.

Prefer container-aware components where component width, not viewport width, determines behavior.

## 15. Typography

### 15.1 Typography objectives

Typography must provide:

- legibility;
- hierarchy;
- brand character;
- compactness or openness appropriate to the product;
- support for required scripts and languages;
- stable rendering and performance.

### 15.2 Typeface selection

Evaluate:

- small-size clarity;
- numeral quality;
- punctuation;
- weight range;
- variable font support;
- language coverage;
- licensing;
- load cost;
- tabular figures;
- code glyph distinction;
- tone.

Do not select a typeface solely because it is trendy.

### 15.3 Type roles

Define roles such as:

- display;
- page title;
- section title;
- subsection title;
- body;
- supporting body;
- label;
- caption;
- data value;
- code;
- button;
- navigation.

Specify font size, line height, weight, letter spacing, and responsive behavior.

### 15.4 Hierarchy discipline

Use differences in this order when possible:

1. placement and spacing;
2. weight;
3. size;
4. color;
5. decorative treatment.

Do not create hierarchy solely by making everything larger.

### 15.5 Numbers and data

For financial, analytical, or tabular interfaces:

- use tabular numerals where alignment matters;
- align decimals when comparison requires it;
- include units;
- distinguish estimated, pending, and final values;
- avoid ambiguous abbreviations;
- localize separators and currency placement.

### 15.6 Text scaling

Support browser zoom and platform text scaling without clipping, overlap, hidden controls, or forced horizontal scrolling for ordinary content.

## 16. Color

### 16.1 Color roles

Define color by function:

- canvas;
- surface;
- raised surface;
- primary text;
- secondary text;
- disabled text;
- border;
- accent;
- focus;
- selection;
- success;
- warning;
- error;
- information;
- data series.

### 16.2 Contrast

Target WCAG 2.2 AA minimums for text and non-text contrast. Test actual rendered combinations, not token names.

Do not rely on opacity alone for secondary text without verifying contrast on every underlying surface.

### 16.3 Color independence

Never use color as the only indicator of:

- error;
- success;
- selection;
- chart series;
- required status;
- increase or decrease;
- risk level.

Pair color with text, icon, pattern, position, or shape.

### 16.4 Status semantics

Status colors must remain consistent across the product. Do not use the same red for both destructive actions and ordinary brand emphasis.

### 16.5 Data visualization

- use color to encode a limited number of meaningful distinctions;
- avoid rainbow palettes for ordered data;
- maintain category identity across charts;
- support color-vision deficiency;
- label directly where possible;
- avoid misleading gradients and decorative 3D effects.

## 17. Iconography, Imagery, and Illustration

### 17.1 Icons

Icons must communicate a real object, action, status, or navigation concept.

- pair unfamiliar icons with labels;
- use one coherent icon family or a deliberately harmonized set;
- maintain consistent optical size, stroke, corner language, and filled/outlined logic;
- provide accessible names for icon-only controls;
- do not use icons as decoration beside every heading;
- avoid ambiguous icons for destructive or consequential actions.

### 17.2 Images

Use images when they improve comprehension, trust, emotional context, product demonstration, or brand expression.

- use the correct aspect ratio and crop behavior;
- define focal points;
- provide meaningful alternative text or mark decorative images correctly;
- reserve dimensions to prevent layout shift;
- adapt art direction across breakpoints;
- avoid unrelated stock photography.

### 17.3 Illustration

Illustration should have a defined role:

- explain a concept;
- orient the user;
- support onboarding;
- make an empty state recognizable;
- reinforce brand.

Do not let illustration overpower the next action.

## 18. Components as Contracts

Every reusable component must define:

- purpose;
- when to use;
- when not to use;
- anatomy;
- variants;
- sizes or density;
- content rules;
- default, hover, pressed, focus, disabled, loading, selected, error, and success states as applicable;
- keyboard behavior;
- pointer and touch behavior;
- screen-reader behavior;
- responsive behavior;
- overflow and truncation behavior;
- internationalization behavior;
- theming behavior;
- data and event API;
- testing requirements.

### 18.1 Prefer semantic native controls

Use native semantic elements when they provide correct behavior. Recreate them only when necessary, preserving keyboard, focus, naming, and state semantics.

### 18.2 Component boundaries

Create a component when one or more are true:

- the pattern repeats;
- behavior is complex;
- accessibility logic should be centralized;
- the visual contract must remain consistent;
- it has multiple stable variants.

Do not componentize every wrapper or one-off layout fragment.

### 18.3 State preservation

Components must not unexpectedly lose user input or selection when rerendered, reordered, filtered, or switched between modes.

Read `references/component-contracts.md`.

## 20. Forms and Data Entry

### 20.1 Form structure

- group fields by user goal, not database schema;
- use a single column by default;
- use multiple columns only for strongly related short fields;
- keep labels persistent;
- indicate optionality consistently;
- place help near the field it explains;
- use appropriate input types and autocomplete attributes;
- preserve entered data after errors;
- avoid asking for information already known or previously entered.

### 20.2 Validation timing

- explain format before entry when constraints are unusual;
- validate syntax after sufficient input or on blur;
- validate cross-field rules when all required values exist;
- validate again on submit;
- do not show errors before the user has had a reasonable chance to act.

### 20.3 Error messages

A good error message states:

1. what is wrong;
2. where it is wrong;
3. how to fix it;
4. what was preserved;
5. whether the action can be retried.

Use specific language. Avoid blame and internal error codes as the only explanation.

### 20.4 Multi-step forms

- show progress in meaningful terms;
- allow safe backward navigation;
- autosave when appropriate;
- avoid splitting trivial forms into artificial steps;
- provide a review step for consequential submissions;
- show what can still be edited;
- explain what happens after submission.

### 20.5 Authentication

- support password managers and paste;
- do not require cognitive-function tests such as memorizing or transcribing a password when accessible alternatives exist;
- explain why verification is required;
- preserve intended destination after authentication;
- provide recovery paths.

---

## 21. Search, Filtering, Sorting, and Large Collections

### 21.1 Search

Define:

- searchable fields;
- query parsing;
- typo tolerance;
- result categories;
- ranking logic if relevant;
- recent searches;
- suggestions;
- keyboard interaction;
- empty results;
- loading and partial results;
- privacy implications.

### 21.2 Filters

- use labels users understand;
- show active filters and count;
- support clear-all;
- preserve filter state when navigating to an item and back;
- distinguish inclusive from exclusive filters;
- show result count when feasible;
- avoid hiding crucial filters behind an unlabeled icon;
- use mobile sheets or dedicated filter views when space is constrained.

### 21.3 Sorting

Use sort criteria that correspond to real user decisions. Clearly indicate current sort and direction.

### 21.4 Pagination versus continuous loading

Use pagination when users need stable location, comparison, return, or a sense of total size. Use continuous loading for discovery feeds where exact location is less important. Provide accessible loading, position restoration, and an alternative to endless scrolling where appropriate.

---

## 22. Tables and Dense Data

### 22.1 Use a table when comparison is the task

Do not convert structured comparative data into cards merely to appear modern.

### 22.2 Table requirements

Consider:

- descriptive caption or context;
- sticky header;
- column priority;
- sorting;
- filtering;
- resizing;
- pinning;
- row selection;
- bulk actions;
- pagination or virtualization;
- loading rows;
- empty state;
- inline editing;
- validation;
- keyboard navigation;
- screen-reader semantics;
- responsive alternatives;
- export.

### 22.3 Alignment

- align text left in left-to-right languages unless the language or data requires otherwise;
- align numbers for comparison;
- align actions consistently;
- keep units visible;
- avoid center alignment for dense data.

### 22.4 Responsive behavior

Do not blindly stack every row into a card. Alternatives include:

- horizontal scrolling with frozen priority columns;
- column chooser;
- progressive disclosure;
- summary row leading to detail view;
- responsive priority hiding;
- dedicated mobile object list.

---

## 23. Dashboards and Data Visualization

### 23.1 Begin with decisions, not charts

For every widget, define the question it answers and the action it supports.

Examples:

- What changed since yesterday?
- Which accounts need attention?
- Are we within target?
- Where is the bottleneck?
- What caused the anomaly?

Remove widgets that do not support a decision.

### 23.2 Hierarchy

A dashboard should normally provide:

1. status and urgent exceptions;
2. key trend or outcome;
3. explanatory breakdown;
4. detailed records;
5. controls and time range.

### 23.3 Chart selection

- line: trend over ordered time;
- bar: category comparison;
- stacked bar: composition with comparison;
- scatter: relationship and outliers;
- histogram: distribution;
- box plot: distribution summary;
- heatmap: matrix intensity;
- area: cumulative volume when overlap does not mislead;
- pie or donut: only for a small, simple part-to-whole relationship where precise comparison is not required.

### 23.4 Chart integrity

- use honest scales;
- label units;
- distinguish missing from zero;
- show uncertainty where applicable;
- avoid 3D decoration;
- do not truncate axes in ways that misrepresent magnitude without clear reason;
- make hover information available through keyboard or visible alternatives;
- provide a table or textual summary when needed for accessibility.

---

## 24. Mobile and Touch Design

### 24.1 Reach and ergonomics

Place frequent actions within comfortable reach when possible. Keep destructive actions away from accidental activation zones.

### 24.2 Target size

WCAG 2.2 AA requires a minimum pointer target of 24 by 24 CSS pixels with defined exceptions. For touch interfaces, larger targets are usually appropriate; use platform guidance and user context, commonly around 44–48 logical pixels for comfortable primary controls.

### 24.3 Safe areas and system UI

Account for:

- notches and cutouts;
- home indicators;
- status bars;
- navigation bars;
- browser chrome;
- foldables;
- orientation changes;
- virtual keyboard;
- dynamic text size.

### 24.4 Gestures

- use familiar gestures;
- provide visible alternatives for nonessential gestures;
- provide a single-pointer alternative to dragging when dragging is not essential;
- do not make hidden gestures the only path to critical actions;
- avoid gesture conflicts near system edges.

### 24.5 Mobile navigation

Bottom navigation is suitable for a small number of primary destinations. Drawers are appropriate for less frequent or larger navigation sets. Avoid using a hamburger menu to hide the product's core task.

### 24.6 Mobile forms

- use correct keyboard types;
- avoid inputs smaller than comfortable text size;
- keep the active field and error visible above the keyboard;
- support autofill and password managers;
- reduce typing through choices, scanning, defaults, and stored data where safe.

---

## 25. Desktop and Professional Software

### 25.1 Spatial stability

Frequent users build spatial memory. Avoid unnecessary movement of navigation, toolbars, status, and primary panels.

### 25.2 Keyboard operation

Professional software should define:

- full tab and arrow navigation where appropriate;
- common shortcuts;
- discoverable shortcut labels;
- command palette where useful;
- focus management;
- shortcut conflict handling;
- user customization when justified.

### 25.3 Panels and workspaces

For complex tools, consider:

- collapsible panels;
- resizable split panes;
- remembered layout;
- reset layout;
- multiple workspace presets;
- visible active context;
- no accidental loss of unsaved work.

### 25.4 Menus

Use established desktop menu structures for application-wide commands. Keep object-specific actions contextual. Do not overload toolbars with low-frequency actions.

### 25.5 Dense controls

Compact controls are acceptable for expert tools when hit areas, focus, labels, and accessibility remain adequate. Density is not the same as clutter.

## 26. Creative Tools, Editors, and Canvases

### 26.1 Canvas dominance

The work product should remain visually primary. UI chrome must support, not compete with, the canvas.

### 26.2 Selection model

Make these states unmistakable:

- nothing selected;
- one object selected;
- multiple objects selected;
- parent/group selected;
- locked object;
- hidden object;
- hover target;
- drag target;
- drop location;
- editing text versus moving an object.

### 26.3 Non-destructive editing

Prefer:

- layers;
- history;
- undo and redo;
- snapshots or versions;
- reversible effects;
- duplicate before destructive conversion;
- clear export versus source distinction.

### 26.4 Tool behavior

Each tool must define:

- activation;
- cursor;
- direct-manipulation behavior;
- modifier keys;
- constraints;
- snapping;
- cancel behavior;
- confirmation if destructive;
- status or hint text;
- touch alternative where supported.

### 26.5 Zoom and navigation

Support:

- fit to content;
- fit selection;
- actual size;
- zoom controls;
- keyboard and wheel behavior;
- panning;
- minimap where useful;
- reset view.

### 26.6 Export

Show:

- format;
- dimensions;
- quality;
- file size estimate when feasible;
- transparency;
- naming;
- destination;
- progress;
- failure recovery;
- what remains editable afterward.

## 27. Developer Tools and IDEs

### 27.1 Core regions

A typical IDE may include:

- activity or mode switcher;
- file explorer;
- editor tabs;
- editor groups;
- terminal or output;
- problems;
- source control;
- search;
- debug controls;
- status bar;
- command palette;
- AI panel.

Do not include every region by default. Prioritize the user's workflow.

### 27.2 Editor quality

Define:

- syntax and semantic highlighting;
- selection and cursor contrast;
- line numbers;
- minimap choice;
- diagnostics;
- inline hints;
- hover information;
- diff states;
- merge conflict states;
- read-only files;
- large file behavior;
- split layout;
- focus mode.

### 27.3 AI-assisted coding

AI features must show:

- what context is being used;
- what files may change;
- planned actions;
- command execution requests;
- diffs before acceptance when appropriate;
- progress;
- tool errors;
- cancellation;
- rollback;
- provenance of generated changes;
- permission boundaries.

Do not make an AI chat panel cover the code or obscure the changes it proposes.

## 28. AI Product Interaction Patterns

### 28.1 Capability clarity

Explain what the system can do, what inputs it uses, and what consequential actions require approval.

### 28.2 Prompt input

Support:

- multiline input;
- attachments or context where available;
- example prompts appropriate to the current state;
- history;
- editable previous prompts;
- stop generation;
- retry;
- model or mode choice only when it changes meaningful behavior.

### 28.3 Streaming

While generating:

- show active status;
- preserve reading stability;
- allow stop;
- distinguish partial output from complete output;
- avoid forcing the viewport to the bottom when the user scrolls up;
- announce status accessibly without excessive interruptions.

### 28.4 Tool use and actions

For external actions, show:

- action intent;
- target;
- required permission;
- parameters that matter;
- current progress;
- success or failure;
- side effects;
- undo or recovery where possible.

### 28.5 Uncertainty and provenance

- distinguish sourced facts, inference, generated content, and user-provided content;
- expose citations or source references when the task requires factual trust;
- do not use fabricated confidence percentages;
- state uncertainty in useful language;
- support correction and regeneration.

### 28.6 Branching and versions

For substantial outputs, support:

- alternatives;
- version comparison;
- restore;
- continue from a version;
- edit before applying;
- clear ownership of final changes.

## 29. Content Design and Microcopy

### 29.1 Voice

Define voice using useful traits tied to behavior, for example:

- direct, not abrupt;
- confident, not absolute;
- technical, not obscure;
- friendly, not childish;
- concise, not incomplete.

### 29.2 Button labels

Use verb-led labels describing the result:

- “Create project” rather than “Submit”;
- “Save changes” rather than “OK”;
- “Delete invoice” rather than “Yes”.

### 29.3 Headings

Headings should communicate the task or content, not merely decorate the page.

### 29.4 Empty states

An empty state should explain:

- what belongs here;
- why it is empty when relevant;
- the most useful next action;
- any prerequisites.

Do not use celebratory illustrations for serious failure or permission states.

### 29.5 Confirmation and destructive copy

State:

- the exact object affected;
- permanence or reversibility;
- related effects;
- safer alternative where available;
- clear destructive action label.

### 29.6 Notifications

Notifications should be timely, actionable, and controllable. Avoid notifying users about events that are neither important nor actionable.

---

## 30. Motion and Interaction Feedback

### 30.1 Motion purpose

Every motion should serve one or more purposes:

- show cause and effect;
- preserve spatial continuity;
- confirm input;
- communicate state change;
- direct attention;
- indicate progress;
- support brand character without obscuring tasks.

### 30.2 Timing

Use short durations for direct manipulation and simple state changes; longer durations may be appropriate for large spatial transitions. Test perceived responsiveness rather than relying on fixed numbers.

### 30.3 Performance

Prefer compositor-friendly properties such as transform and opacity. Avoid unnecessary layout-triggering animation in repeated or large elements.

### 30.4 Reduced motion

Respect `prefers-reduced-motion` and platform settings. Replace large movement, parallax, auto-zoom, and repeated pulsing with fades, instant changes, or reduced distance where appropriate.

### 30.5 Loading feedback

Choose based on duration and structure:

- immediate state change for instant actions;
- button progress for local actions;
- skeleton for predictable content structure;
- determinate progress when progress is measurable;
- stage-based progress for multi-step work;
- cancel and background continuation for long operations where technically possible.

Do not show a fake precise percentage when progress cannot be measured.

---

## 31. Accessibility Requirements

Target WCAG 2.2 AA for web unless the product requires more.

### 31.1 Semantics

- use correct landmarks and heading hierarchy;
- use buttons for actions and links for navigation;
- use labels, names, roles, and states correctly;
- avoid unnecessary ARIA when native semantics suffice;
- provide status announcements for meaningful asynchronous updates.

### 31.2 Keyboard

- all functionality must be keyboard operable unless inherently impossible;
- focus order must match logical task order;
- focus must be visible;
- focused elements must not be obscured by sticky content;
- dialogs must manage focus and return it appropriately;
- no keyboard traps;
- common composite widgets should follow established keyboard patterns.

### 31.3 Pointer and touch

- provide alternatives to multipoint or path-based gestures where possible;
- provide alternatives to dragging when dragging is not essential;
- support pointer cancellation;
- ensure adequate target size and spacing;
- avoid hover-only functionality.

### 31.4 Visual

- verify text and non-text contrast;
- support text resize and zoom;
- do not communicate by color alone;
- preserve meaning in forced-colors or high-contrast modes where applicable;
- provide visible focus;
- avoid images of text except where essential.

### 31.5 Cognitive

- use plain, consistent language;
- avoid unnecessary time limits;
- provide consistent help;
- do not require re-entry of information already provided in the same process unless necessary;
- support accessible authentication;
- reduce surprise and context changes.

### 31.6 Media

Provide captions, transcripts, audio description, controls, and non-autoplay behavior as required by the media and conformance target.

### 31.7 Testing

Accessibility validation must include:

- automated scanning;
- keyboard-only review;
- screen-reader review for critical flows;
- zoom and text scaling;
- contrast testing;
- reduced-motion testing;
- high-contrast or forced-colors testing where relevant;
- touch-target review;
- error and authentication review.

Read `references/accessibility.md`.

---

## 32. Internationalization and Localization

Design for:

- text expansion;
- long words;
- right-to-left layout;
- local number, date, time, and currency formats;
- plural rules;
- name and address variability;
- different writing systems;
- locale-specific sorting;
- translated imagery and screenshots;
- regional legal content;
- time zones.

Do not concatenate translated fragments to form sentences. Do not hard-code field assumptions that apply only to one country.

Mirroring should follow semantic direction. Some content—such as media playback timelines, charts, maps, code, or brand marks—may not mirror automatically.

## 33. Privacy, Trust, and Ethical Design

### 33.1 Data minimization

Ask only for data necessary for the current purpose. Explain why sensitive data is needed and how it will be used.

### 33.2 Consent

Consent must be informed, specific, and freely given where required. Do not use visual manipulation, preselected options, hidden decline paths, or misleading hierarchy.

### 33.3 Permissions

Request permissions at the moment of value, with context. Provide a useful path when permission is denied.

### 33.4 Sensitive information

- mask only when appropriate;
- allow deliberate reveal;
- avoid exposing sensitive data in notifications, URLs, logs, or shared screens;
- define timeout and session behavior;
- communicate data freshness and source.

### 33.5 Avoid dark patterns

Never use:

- disguised advertisements;
- confirmshaming;
- forced continuity;
- hidden fees;
- difficult cancellation;
- trick questions;
- visual interference that makes refusal hard to find;
- fake scarcity;
- unauthorized default opt-in;
- deceptive progress.

## 34. Design-to-Code Workflow

When implementing or modifying code, follow this order.

### 34.1 Inspect first

Before editing:

- inspect project structure;
- identify framework and versions;
- locate routing, state, components, styles, assets, tests, and design tokens;
- identify installed dependencies;
- run or inspect the existing app when possible;
- find existing patterns to preserve or improve;
- determine responsive and accessibility conventions;
- check build and lint scripts.

Do not introduce a new framework, state library, icon library, or styling system when the existing stack can support the requirement.

### 34.2 Preserve functionality

A redesign must not remove working features, permissions, validation, analytics, or data states unless explicitly required.

### 34.3 Implement architecture

- keep components cohesive;
- separate data, behavior, and presentation where it improves maintainability;
- centralize tokens;
- avoid arbitrary one-off values;
- maintain semantic HTML;
- support server and client boundaries correctly;
- avoid unnecessary global state;
- define error boundaries and loading behavior;
- use stable keys and state preservation.

### 34.4 Use real content structure

Implement realistic labels, values, and edge cases. Do not leave critical UI dependent on lorem ipsum.

### 34.5 Responsive implementation

Test at:

- narrow mobile;
- common mobile;
- tablet portrait and landscape where relevant;
- standard laptop;
- wide desktop;
- high zoom;
- long translated content.

### 34.6 Accessibility implementation

Implement semantics, focus, keyboard behavior, names, descriptions, errors, live regions, and reduced motion as part of the component—not as comments for later.

### 34.7 Performance

- optimize images and fonts;
- reserve media dimensions;
- avoid unnecessary client JavaScript;
- virtualize only when data volume requires it;
- debounce or defer expensive operations appropriately;
- prevent layout shift;
- avoid animation that causes repeated layout;
- measure rather than guess.

### 34.8 Verification

Run available:

- build;
- type checking;
- lint;
- unit tests;
- component tests;
- end-to-end tests;
- accessibility checks;
- visual regression checks.

Then inspect the rendered interface, not only the source code.

Read `references/design-to-code.md`.

## 35. Screenshot, Reference, and Existing-UI Analysis

When given screenshots or references:

1. Identify layout regions.
2. Identify visual hierarchy.
3. Identify typography roles.
4. Infer spacing and grid.
5. Identify component patterns.
6. Identify interaction states not visible in the screenshot.
7. Identify accessibility risks.
8. Separate transferable principles from brand-specific details.
9. Reconstruct behavior, not only appearance.
10. Adapt to the current product rather than cloning.

When redesigning an existing product, preserve users' learned mental models unless there is a clear reason to change them.

## 36. Critique Method

Critique in this order:

1. Product and user fit.
2. Task flow and information architecture.
3. Content clarity.
4. State completeness.
5. Accessibility.
6. Responsive behavior.
7. Interaction feedback.
8. Visual hierarchy.
9. Consistency and system quality.
10. Polish and distinctiveness.

Do not begin critique with subjective color preference when users cannot complete the task.

For each issue, provide:

- evidence;
- impact;
- severity;
- recommended change;
- expected outcome;
- verification method.

Severity levels:

- **Blocker:** prevents task completion, causes unsafe action, or excludes a major user group.
- **High:** substantial confusion, repeated error, serious accessibility failure, or major inconsistency.
- **Medium:** slows users, weakens comprehension, or creates avoidable friction.
- **Low:** polish, consistency, or minor efficiency issue.

## 37. Self-Audit and Revision Loop

The agent must not treat the first complete output as final.

### 37.1 Five-second comprehension test

Without reading detailed copy, can a user answer:

- What product or area is this?
- Where am I?
- What is the main information?
- What can I do?
- What should I do next?

### 37.2 Grayscale hierarchy test

Remove color mentally or visually. Hierarchy and states should remain understandable.

### 37.3 Blur test

At a blurred glance, the major regions and focal point should remain clear.

### 37.4 Edge-case test

Test:

- longest realistic label;
- smallest and largest value;
- no data;
- one item;
- thousands of items;
- missing image;
- slow network;
- failed request;
- expired session;
- keyboard-only operation;
- 200% zoom;
- narrow screen;
- right-to-left content.

### 37.5 Genericness test

Identify any element that exists because it is common in AI-generated designs rather than because the product needs it. Remove or justify it.

### 37.6 Implementation test

Confirm that every proposed interaction can be implemented in the selected stack and data model. Flag dependencies and backend requirements.

### 37.7 Revision rule

If any mandatory quality dimension scores below threshold, revise before delivery.

## 39. Mandatory Final Quality Gates

Before declaring work complete, verify:

### Product and UX

- [ ] The primary user and task are explicit.
- [ ] The design is specific to the product archetype.
- [ ] Information architecture matches the object and task model.
- [ ] Critical flows include success, failure, and recovery.
- [ ] Navigation communicates location and available destinations.

- [ ] No major dead ends exist.

### Visual system

- [ ] One focal point exists per screen or decision region.
- [ ] Typography roles are consistent.
- [ ] Spacing communicates relationships.
- [ ] Color has semantic logic.
- [ ] Icons are understandable and coherent.
- [ ] The interface avoids generic AI visual defaults.
- [ ] Dark and light themes are independently validated if included.

### Interaction

- [ ] Interactive elements have required states.
- [ ] Feedback is immediate and proportional.
- [ ] Destructive actions are protected appropriately.
- [ ] Long operations show meaningful progress and recovery.
- [ ] Unsaved work is protected.
- [ ] Keyboard, pointer, touch, and assistive operation are considered.

### Accessibility

- [ ] Semantics are correct.
- [ ] Contrast is verified.
- [ ] Focus is visible and not obscured.
- [ ] Keyboard operation is complete.
- [ ] Touch targets are adequate.
- [ ] Color is not the only signal.
- [ ] Text resize and zoom work.
- [ ] Reduced motion is supported.

- [ ] Large datasets have a performance strategy.
- [ ] Images and fonts are optimized.
- [ ] Build, lint, types, and tests pass where available.

## 43. Behavior Under Ambiguity

When requirements are incomplete:

1. Preserve explicit constraints.
2. Infer from product archetype and common user goals.
3. Choose conservative, reversible defaults.
4. Mark assumptions that materially affect the result.
5. Avoid blocking progress for minor missing details.
6. Do not invent facts or business rules.
7. Design the architecture so uncertain details can change without rewriting the entire interface.

When two interpretations produce materially different products, present the chosen interpretation and briefly note the alternative.

---

## 44. Behavior When Editing Existing Code

- inspect before replacing;
- reuse existing components and tokens where they are sound;
- preserve routes, state, data fetching, validation, and accessibility behavior;
- do not erase complex functionality to make a screenshot look cleaner;
- keep changes scoped and reviewable;
- avoid giant component rewrites unless architecture requires it;
- show or summarize the important design and implementation decisions;
- verify regressions.

---

## 45. Behavior When the User Requests “Premium”

Translate “premium” into operational qualities:

- precise alignment;
- disciplined spacing;
- restrained visual noise;
- high-quality typography;
- coherent materials and elevation;
- strong content hierarchy;
- deliberate motion;
- realistic content;
- excellent loading, empty, error, and success states;
- no rough implementation edges;
- no generic decorative trends without product justification.

Premium does not automatically mean black backgrounds, gold accents, glass cards, excessive whitespace, or slow animation.

---

## 46. Behavior When the User Requests “Minimal”

Minimal means low unnecessary complexity, not missing functionality.

- remove low-value elements;
- consolidate related actions;
- preserve essential context;
- reveal advanced controls progressively;
- use typography and spacing rather than excessive containers;
- maintain clear error, help, and recovery paths.

Do not hide everything behind icons or menus merely to reduce visual elements.

---

## 47. Behavior When the User Requests “Modern”

Modern means current interaction quality, responsiveness, accessibility, and technical execution—not automatic use of gradients, glass, bento grids, or oversized rounded corners.

Select contemporary techniques only when they support the product.

---

## 48. Behavior When the User Requests a Clone

- identify the reference product's useful principles;
- preserve the current product's own identity and needs;
- avoid copying proprietary visual assets, copy, or exact layout;
- state which interaction conventions are being adapted;
- improve known weaknesses instead of reproducing them.

---

## 49. Failure Conditions

The design is not complete if any of these are true:

- the primary task is unclear;
- content is mostly filler;
- only the happy path exists;
- mobile is a compressed desktop layout;
- keyboard users cannot complete the task;
- focus is invisible or obscured;
- controls rely on color or hover alone;
- loading and errors are unspecified;
- destructive actions lack appropriate safeguards;
- the visual style could belong to any product;
- implementation requires unspecified backend capabilities;
- data visualizations are misleading;
- the design uses fake trust claims;
- state disappears unexpectedly;
- the interface was not inspected after implementation.

## 51. Final Directive

Do not optimize for the appearance of design competence. Perform the design work.

Understand the product. Model the user. Structure the information. Specify the behavior. Create a coherent visual system. Handle states and failure. Implement accessibly. Inspect the rendered result. Test it. Score it. Revise it.

The final interface should feel inevitable for its purpose, not merely fashionable.
